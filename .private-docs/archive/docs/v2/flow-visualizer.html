Event Kind Reference

Five event kinds across two independent protocols. Each validated on its own terms.

1776
PMA
Prepared Migration Authority. Root enrollment event signed by the old key. Anchored via NIP-03 OpenTimestamps.
1779
PMU
Authority Update. Rotates the migration key. Requires detached signatures from the old key and the new migration key.
1777
PMX
Migration Execution. Performs the migration. Signed by the active migration key with detached consent from the successor.
1778
STC
Social Transition Claim. Immutable statement by old or new key that a transition exists. Not cryptographic proof.
31778
STA
Social Transition Attestation. Addressable event expressing an attestor's current stance. Latest version wins.
Validate separately. Present together if useful.
Core design principle
NIP-XX

Prepared Migration

Cryptographic. Deterministic. Every client reaches the same conclusion given the same events.

Requires: NIP-01, NIP-03
Events: 1776, 1779, 1777
NIP-XY

Social Transition

Social. Viewer-relative. The same events produce different results depending on who is looking.

Requires: NIP-01
Events: 1778, 31778
NIP-XX · Prepared Migration

Happy Path

The simplest scenario: user enrolls during calm times, then executes migration when needed later.

Phase 1 — Enrollment (calm times)
Old Key (O) kind:1776 Publishes PMA naming migration key M signed by O
OTS / Bitcoin kind:1040 PMA receives OpenTimestamps proof at block height H anchor_height = H
Crisis — Key lost or compromised
O Old key is no longer safe — but M is still held securely
Phase 2 — Execution
N (successor) Provides detached consent signature (ns) over the PMX preimage proves N is live
M (migration) kind:1777 Publishes PMX: o=O, n=N, e=PMA_id, ns=consent signed by M
Phase 3 — Client Resolution
Client Queries kind:1776, kind:1040, kind:1777 for pubkey O from relays
Client Validates PMA → checks OTS proof → validates PMX deterministic
Result State: prepared_migrated — successor is N ✓
NIP-XX · Prepared Migration

Authority Update Chain

User rotates the migration key before needing to execute. Requires three-party consent: old key, current migration key, and the next migration key.

Initial enrollment
Old Key (O) kind:1776 PMA: m = M1 + OTS at block H1
Authority Update (M1 → M2)
M1 (current) Builds PMU event, collects detached signatures coordinator
O Signs os: detached signature over canonical preimage old key consent
M2 (next) Signs ns: detached signature over same canonical preimage new key consent
M1 kind:1779 Publishes PMU: e=PMA_id, u=M2, os, ns + OTS at block H2
Later execution uses M2
M2 kind:1777 PMX now references the PMU, not the original PMA M1 cannot execute

AUTHORITY CHAIN RESOLUTION

Step 1
Find valid PMA roots
Collect all valid kind:1776 events authored by O. Compute anchor_height for each.
Step 2
Choose canonical root
Select the PMA with minimum anchor_height. If tied, state is conflicting_roots.
Step 3
Walk PMU chain
Starting at the canonical root, follow PMU events linearly. Any fork = conflicting_authority_updates.
Step 4
Resolve active migration key
The last event in the chain (PMA or PMU) determines the active migration key.
NIP-XX · Prepared Migration

Terminal States

Six possible outcomes. Three are clean, three are conflicts. Clients MUST NOT auto-resolve conflicts.

none
No valid PMA exists for this pubkey. No prepared migration has been enrolled.
Client behavior Nothing to show. Profile is normal.
prepared_enrolled
A valid authority chain exists but no PMX has been published. The migration key is standing by.
Client behavior Show enrollment badge. "This identity has a prepared migration backup."
prepared_migrated
A valid PMX names exactly one successor key N. The cryptographic migration is complete.
Client behavior Show migration result. Offer: follow N, follow both, mute old key, inspect evidence.
conflicting_roots
Two or more valid PMAs share the same minimum anchor_height. Cannot determine canonical root.
Client behavior Show conflict warning. Do NOT pick a winner. Do NOT use created_at, event id, or relay arrival time.
conflicting_authority_updates
The PMU chain forks: multiple PMUs reference the same parent authority event.
Client behavior Show conflict warning. Authority chain is ambiguous.
conflicting_executions
Valid PMX events reference the active authority but name different successor keys.
Client behavior Show conflict warning. Multiple successor claims exist.
NIP-XY · Social Transition

Claim & Attestation Flow

Social signaling for identity transition. Claims are immutable. Attestations are mutable (latest stance wins).

Old Key (O) kind:1778 STC with r="old": "I moved to N" immutable

Friend Alice kind:31778 STA: s="support", m="in_person" addressable

Because kind:31778 is addressable, only the latest valid STA per attestor per transition counts. Previous stances are superseded. Public stance history may be lost — this is an intentional tradeoff.

Scenarios

Real-world situations showing how both protocols interact under different conditions.

social-only Unprepared User — Lost Key
▾

User lost their old key and never enrolled in Prepared Migration. Recovery relies entirely on social evidence.

Old key (O) is lost
User creates a new key N from scratch
kind:1778 N publishes STC (r="new")
"Lost my old key, this is my new one"
O cannot publish an STC (key is lost)
Only the N-side claim exists
Friends verify out-of-band and publish STAs
kind:31778 s="support", m="video" / m="in_person"
Client resolution
NIP-XX: none (no PMA exists)
NIP-XY: socially_supported (2 followed supporters)
UI: "N claims to be O's successor. 2 people you follow confirmed. No prepared migration exists."
crypto Key Compromise Race
▾

Attacker compromises O and tries to create a fake migration. But the user had enrolled a PMA earlier. The earliest Bitcoin anchor wins.

User enrolled PMA at block H1
Migration key M is stored securely offline
Attacker compromises O
Attacker has the old key but not M
Attacker publishes fake STC using O
kind:1778 r="old", n=A_new (attacker's key)
Attacker tries to create competing PMA
kind:1776 Using compromised O, OTS anchors at block H2
User executes real migration with M
kind:1777 References original PMA, names real successor N
Client resolution
NIP-XX: 2 PMAs found. User's anchor H1 < attacker's H2
User's PMA is canonical. State: prepared_migrated → N
NIP-XY: Attacker's STC exists but is independent — doesn't affect NIP-XX
Key insight: The earliest Bitcoin timestamp determines the canonical authority. The attacker's later PMA has higher anchor_height and is simply not the canonical root. Social claims from the attacker are shown separately and do not affect the cryptographic result.
both Disputed Transition (Social Split)
▾

The community disagrees about a transition's legitimacy. Viewer's contacts are split.

N publishes STC (r="new")
Claims to be O's successor
Friend A: STA s="support", m="voice"
"I verified over a call"
Friend B: STA s="oppose"
"This doesn't match what I know about O"
Client resolution
NIP-XX: none (no PMA)
NIP-XY: socially_split (follow set includes support + oppose)
UI: "Disputed transition. 1 contact supports, 1 opposes. No cryptographic migration exists. [View evidence] [Follow N] [Ignore]"

Rule: Client MUST NOT auto-resolve. User decides.
tension Crypto Valid, Socially Opposed
▾

A prepared migration is cryptographically valid, but the social layer has opposition. This tests the "validate separately" principle.

Valid PMA + PMX chain exists
NIP-XX: prepared_migrated → N
Followed contacts publish "oppose" STAs
"I spoke to O, they say they didn't authorize this"
Client resolution
NIP-XX: prepared_migrated (valid)
NIP-XY: socially_opposed
Key principle: NIP-XY MUST NOT veto NIP-XX. The cryptographic migration is valid on its own terms. Social opposition is surfaced as additional context, not as a revocation. The client shows both layers and lets the user decide how to weigh them.
conflict Conflicting Roots (Same-Block Ambiguity)
▾

Two valid PMAs for the same pubkey are timestamped in the same Bitcoin block. No tiebreaker exists — this is intentional.

PMA #1: o=O, m=M1
OTS anchor at block H
PMA #2: o=O, m=M2
OTS anchor at block H (same block!)
Client resolution: conflicting_roots
min_anchor_height is H, but 2 PMAs share it
No canonical root can be determined
Rule: Clients MUST NOT try to break ties using event id, created_at, relay arrival time, or any other non-standard rule. This ambiguity is a conflict state. The user should be shown the evidence and decide.
